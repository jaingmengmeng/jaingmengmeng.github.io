Zookeeper模拟心跳线

一、实验要求

​ 在Hadoop MapReduce框架下，利用zookeeper实现leader和follower之间的心跳线维护。

二、配置环境

  • Centos: CentOS Linux release 7.8.2003 (Core)
  • Hadoop: 2.7.3
  • Zookeeper: 3.4.13
  • Jdk: jdk-8u144-linux-x64.tar.gz
  • Intellij IDEA: 2019.3.4
  • Maven: 3.3.6

三、编写mapreduce代码

1、选主逻辑：

在每个 map 任务的 setup 阶段，都会先建立一个 zookeeper 的连接，先访问目录 /heartbeat-leader 下，如果不存在该目录，就新建一个 /heartbeat-leader，然后成为 leader；如果已经存在该目录，就成为一个follower。

2、leader轮询：

leader 每隔2s会去访问目录 /heartbeat，获取目录下的所有子目录，并获取它们的值，如果是一个时间戳，则表示这个 follower 刚刚发送过了一个心跳，就输出该 follower 还存活，最后把这个子目录的值设为 0；如果获得的值是 0，则表示这个 follower 已经宕机，刚刚没有发送心跳线来。

3、follower定时写入：

follower 在创建时会根据自己的 task_id 在目录 /heartbeat 下新建一个与自己 task_id 同名的子目录。每隔两秒，follower 往对应的子目录下写入当前的时间戳，也即发送了一个 heartbeat 信号，表示自己是存活状态。并为了模拟宕机状态，follower 会以 0.3 的概率宕机，也即不往对应的子目录下写入当前的时间戳。

四、配置hadoop参数

1、控制map数量

为了控制 map 的个数，在 hdfs 的 /input 目录下新建 16 个文件，每个文件很小，只包含了一两个单词，这样启动 job 时，就会生成 16 个 map 任务。

六、实验感受

本次实验，利用 Hadoop MapReduce 框架和 Zookeeper 模拟了 leader 和 follower 心跳线的维护，follower 数量达到了16，让我更加理解了 leader-follower 这种架构，并且学会了怎么使用 IDEA 和 Maven来编写 mapreduce 框架下的 java 代码，也学会了如果利用 zookeeper 来保存和读取分布式系统中的信息，同时也对与 Hadoop 的原理和机制有了更深的理解，包括 job 概念，container 概念，task 概念，attempt 概念，map task 概念，reduce task 概念。
